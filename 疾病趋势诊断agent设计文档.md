# 皮肤病诊断Agent设计文档（LangGraph版）

**版本**：v1.0  
**日期**：2026-03-30  
**参考流程**：扣子Coze双模型+知识库架构  
**实现框架**：LangGraph（Python）

---

## 1. 核心流程对照

| 扣子节点 | 对应LangGraph节点 | 职责 |
|---------|------------------|------|
| 开始 | `entry_point` | 接收用户输入（图片/文字） |
| **大模型**（豆包1.5·Pro·视觉推理） | `vision_analysis` | **多模态理解**：看懂皮肤病照片，提取症状描述 |
| **知识库检索** | `rag_retrieval` | 检索相似病例、治疗方案 |
| **大模型_1**（豆包1.8·深度思考） | `llm_reasoning` | **深度推理**：综合症状+知识库，做诊断决策 |
| 知识库写入 | `knowledge_feedback` | 优质案例回写入库（可选） |
| 输出/结束 | `compile_output` | 生成最终报告 |

---

## 2. 关键设计：为什么需要两个模型

| | 第一个模型（Vision） | 第二个模型（LLM） |
|--|---------------------|----------------|
| **输入** | 原始图片 + 用户描述 | 结构化症状 + RAG检索结果 |
| **核心能力** | **视觉理解**：识别病灶类型、分布、严重程度 | **医学推理**：诊断逻辑、治疗方案生成 |
| **模型选择** | GPT-4V / Claude 3 / Qwen-VL | GPT-4 / Claude 3.5 / DeepSeek |
| **输出** | 症状结构化描述（给第二个模型用） | 诊断结论 + 建议 |
| **为什么分开** | 视觉模型贵且慢，只做一次；推理模型专注决策 | 避免视觉干扰纯文本推理 |

## 3. 状态定义（State）

### 3.1 输入层
| 字段 | 类型 | 说明 |
|-----|------|------|
| `user_id` | str | 用户唯一标识 |
| `input_type` | 枚举 | 用户上传类型：image / text / both |
| `raw_image` | bytes | 原始图片数据（可选） |
| `user_description` | str | 用户文字描述（可选） |

### 3.2 第一层模型输出（Vision）
| 字段 | 说明 |
|-----|------|
| `symptoms` | 症状列表，如["红色丘疹", "脓疱", "面部弥漫性红斑"] |
| `distribution` | 分布部位描述，如"面颊部、额头，对称分布" |
| `severity` | 严重程度：轻度/中度/重度 |
| `estimated_type` | 初步诊断类型，如"痤疮（炎性）" |
| `visual_confidence` | 视觉分析置信度，0.0-1.0 |
| `reasoning` | 详细分析过程描述 |

### 3.3 RAG层
| 字段 | 说明 |
|-----|------|
| `rag_query` | 自动生成的检索Query |
| `rag_results` | 检索到的相似病例列表 |

### 3.4 第二层模型输出（LLM推理）
| 字段 | 说明 |
|-----|------|
| `diagnosis` | 主要诊断结论 |
| `confidence` | 诊断置信度，0.0-1.0 |
| `differential` | 鉴别诊断列表（2-3个可能） |
| `treatment_plan` | 治疗方案（外用/口服/辅助治疗） |
| `lifestyle` | 生活建议列表 |
| `follow_up` | 复查时间建议 |
| `red_flags` | 危险信号（如有） |
| `referral_needed` | 是否需要转诊 |

### 3.5 输出层
| 字段 | 说明 |
|-----|------|
| `final_output` | 最终用户报告 |
| `should_feedback` | 是否优质案例，值得入库 |

## 4. 节点（Nodes）详细设计

### 4.1 第一层：Vision模型节点

**职责**：多模态视觉分析，看懂皮肤病照片

**模型选择**：GPT-4V / Claude 3 Sonnet / Qwen-VL-Max

**输入**：
- 用户上传的皮肤照片（base64编码）
- 用户文字描述（可选）

**分析维度**：
1. 皮损类型：粉刺/丘疹/脓疱/结节/囊肿/红斑/鳞屑等
2. 分布部位：面部T区/面颊/额头/下颌/躯干等，是否对称
3. 严重程度：轻/中/重（参考标准：轻度<10个，中度10-30个，重度>30个或囊肿）
4. 伴随特征：油腻/干燥/脱屑/色素沉着/瘢痕
5. 初步判断：最可能的诊断

**输出**：结构化JSON，包含症状列表、分布、严重度、初步诊断、置信度、分析推理、是否建议紧急就医

**副作用**：自动生成RAG检索Query（格式：{初步诊断} {严重度} {症状列表}）

---

### 4.2 RAG检索节点

**职责**：调用RAG服务，检索相似病例和治疗方案

**输入**：
- `rag_query`：自动生成的检索Query
- `vision_analysis.estimated_type`：用于过滤疾病类型
- `vision_analysis.severity`：用于过滤严重程度

**检索参数**：
- Top K：5条相似病例
- 过滤条件：疾病类型、严重程度

**输出**：相似病例列表，每条包含：
- 诊断结果
- 治疗方案
- 预后情况
- 恢复周期

---

### 4.3 第二层：LLM推理节点（核心Agent）

**职责**：深度推理，综合视觉分析+知识库，做诊断决策

**模型选择**：GPT-4 / Claude 3.5 Sonnet / DeepSeek-Coder（医学微调版）

**温度设置**：0.1（低温度，保证诊断稳定性）

**输入上下文构成**：
1. 【视觉分析结果】：AI对照片的分析（症状、分布、严重度、初步判断、置信度、推理过程）
2. 【相似病例参考】：历史相似病例的治疗经过和预后（Top 5）
3. 【患者自述】：用户主观感受描述

**诊断原则**：
- 优先考虑常见诊断（奥卡姆剃刀）
- 列出鉴别诊断（2-3个可能）
- 给出置信度（低置信度时建议线下确诊）
- 治疗方案需具体（药物名称、用法、疗程）

**输出**：结构化JSON，包含：
- 主要诊断
- 诊断置信度
- 鉴别诊断列表
- 治疗方案（外用/口服/辅助治疗）
- 生活建议
- 复查时间
- 危险信号（如有）
- 是否需要转诊

---

### 4.4 输出组装节点

**职责**：组装最终输出，判断是否需要知识库反馈

**组装用户报告**：
- 诊断结论
- 诊断置信度
- 治疗方案
- 生活建议
- 复查时间
- 转诊警告（如需要）

**优质案例判定条件**（同时满足）：
- 诊断置信度 > 0.85
- 不需要转诊
- 视觉分析置信度 > 0.8

**输出**：
- `final_output`：最终用户报告
- `should_feedback`：是否值得入库

---

### 4.5 知识库反馈节点（可选副作用）

**职责**：优质案例回写入知识库（异步，不阻塞主流程）

**触发条件**：`should_feedback` 为 true

**入库内容**：
- 案例类型：confirmed_diagnosis
- 视觉分析结果
- LLM诊断结果
- 用户ID
- 时间戳

**执行方式**：异步写入，不等待返回

## 5. LangGraph拓扑结构

```
┌─────────────┐
│   开始输入   │
│ (图片+描述)  │
└──────┬──────┘
       │
       ▼
┌─────────────────┐
│  vision_analysis│  ← 第一层：多模态视觉模型
│  (GPT-4V等)      │     看懂照片，提取症状
└────────┬────────┘
         │
         ▼
┌─────────────────┐
│  rag_retrieval  │  ← 检索相似病例
│  (组长RAG服务)   │
└────────┬────────┘
         │
         ▼
┌─────────────────┐
│  llm_reasoning  │  ← 第二层：深度推理模型
│  (GPT-4等)       │     【核心Agent】诊断决策
└────────┬────────┘
         │
         ▼
┌─────────────────┐
│  compile_output │  ← 组装报告
└────────┬────────┘
         │
         ├──────────────┐
         │              │
         ▼              ▼
    ┌─────────┐   ┌─────────────┐
    │  结束   │   │knowledge_   │  ← 异步入库（可选）
    │ 输出报告 │   │feedback    │
    └─────────┘   │（副作用）   │
                  └─────────────┘
```

**流程说明**：
- 主流程为线性流程：vision_analysis → rag_retrieval → llm_reasoning → compile_output → END
- knowledge_feedback为副作用分支，不影响主流程响应时间
- 支持断点续跑（通过MemorySaver实现状态持久化）

## 6. 模型选型建议

| 节点 | 推荐模型 | 备选 | 选择理由 |
|-----|---------|------|---------|
| **vision_analysis** | GPT-4V | Claude 3 Sonnet, Qwen-VL-Max | 医学图像理解强 |
| **llm_reasoning** | GPT-4 | Claude 3.5 Sonnet, DeepSeek-V2 | 医学推理严谨 |
| **成本优化** | 可替换为国产模型 | 智谱GLM-4V, 文心一言 | 降低API费用 |

---

## 7. 与RAG的对接

| 参数 | 说明 |
|-----|------|
| **调用时机** | 视觉分析后，LLM推理前 |
| **Query生成** | 自动：`{初步诊断} {严重度} {症状列表}` |
| **返回使用** | 注入LLM prompt，作为"相似病例参考" |
| **反馈入库** | 高置信度案例异步回写，扩充RAG库 |

---

## 8. 边界情况处理

| 场景 | 处理策略 |
|-----|---------|
| 仅文字输入（无图片） | 跳过vision_analysis节点，直接基于文字描述生成RAG Query |
| 视觉分析置信度过低 | 在输出中增加"建议上传更清晰照片"提示 |
| RAG检索无结果 | LLM基于视觉分析结果独立做诊断 |
| LLM诊断置信度过低 | 强制设置`referral_needed=true`，建议线下就诊 |
| 检测到危险信号 | 优先提示紧急就医，再给出诊断建议 |

---

## 9. 安全与合规

1. **医疗免责声明**：所有输出必须包含"AI诊断仅供参考，请以专业医生意见为准"
2. **敏感图片处理**：用户上传图片仅用于诊断，不做持久化存储
3. **转诊机制**：置信度低或检测到危险信号时，必须建议线下就医
4. **隐私保护**：用户ID脱敏处理，案例入库时去除可识别个人信息
